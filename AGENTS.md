# Agent Contract for Sympsense 2.0

Этот файл задает обязательные правила работы AI-агента в репозитории.

## 1) Миссия

Поддерживать персональную медицинскую память пациента через `agent-first` процесс:
`inbox -> просмотр документов -> типизация -> извлечение фактов -> обновление базы -> отчет`.

## 2) Текущий продуктовый фокус

- Пользователь кладет файлы в `data/raw/inbox/<batch_id>`.
- Агент в сессии читает и разбирает файлы сам.
- Пользователь не обязан вручную медицински валидировать каждый фрагмент.
- Агент фиксирует уверенность и помечает сомнительные места как `needs_review`.

Важно:
- Не начинать с разработки новых парсеров, если можно извлечь корректно в текущей сессии.
- Сначала извлечение из реальных документов, потом автоматизация повторяющихся мест.

### 2.1) Политика распознавания (обязательно)

- Агент обязан использовать распознавание так, чтобы качество не падало на сканах и image-only PDF.
- Базовый порядок:
  1. Пробовать `text_layer` (быстро и трассируемо, если слой есть).
  2. Если текстового слоя нет или он низкого качества, переключаться на мультимодальное распознавание страниц (`multimodal_vision`) прямо в сессии.
  3. Если после vision-распознавания уверенность низкая, помечать документ как `needs_review`.
- Для локальных файлов действует тот же принцип, что и для изображений, загруженных в чат: если документ читаем глазами, агент должен использовать vision-путь, а не ограничиваться `text_layer`.

## 3) Архитектурные границы

Слои:
1. `data/raw` - исходные документы (`inbox`, `recognized`, `needs_review`, `non_medical`).
2. `data/canonical` - структурированная база фактов (source of truth).
3. `data/derived` - отчеты и производные артефакты.
4. `data/audit` - журнал операций.

## 4) Обязательные сущности (текущий scope)

- `documents` (реестр документов)
- `labs`
- `doctor_conclusions`
- `medications`
- `symptoms_events`
- `recommendations`

## 5) Код и изменения

- Минимальный осмысленный change-set.
- Без дублирования логики.
- Явные имена и type hints на публичных функциях.
- Ошибки не скрывать.
- Все критичные операции фиксировать в `data/audit/logs/*.ndjson`.

## 6) Безопасность и ограничения

- Не выдавать диагноз как клинический вердикт.
- Не назначать лечение.
- Красные флаги формулировать как «обсудить с лицензированным врачом».
- Локальное хранение по умолчанию.

## 7) Definition of done для batch

1. Документы типизированы.
2. Факты извлечены и записаны в canonical.
3. Реестр документов обновлен.
4. Файлы перемещены из inbox по статусам.
5. Пользователь получил понятную сводку в чате.
